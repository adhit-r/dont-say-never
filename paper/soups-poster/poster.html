Question

When developers write security rules for AI coding agents, does wording matter?

Prompt-engineering folklore says negative rules such as NEVER use eval() can backfire by making the forbidden API more salient. We tested whether this holds for LLM coding agents.

Short answer: rules help. Prohibition wording itself is not the main problem.

Study Design

6 models: 3 GPT/Codex + 3 Claude
6 security prompts across 4 CWE classes
2,160 valid trials, zero final errors

Conditions

  • Control: task prompt only.
  • Negative rule: prohibit insecure pattern.
  • Positive rule: suggest safer alternative.

Security classes

CWE-94 code execution, CWE-328 weak hashing, CWE-319 plaintext HTTP, CWE-338 insecure randomness.

Main Result: Rules Reduce Vulnerable Code

Every model generated fewer vulnerable outputs when a relevant security rule was present.

Polarity Does Not Generalize

Positive framing is not consistently safer than prohibition framing. The direction changes by model and prompt.

Red: positive framing produced more vulnerable code. Blue: positive framing produced less.

Takeaways

Add rules for known CWE classes. Rule presence is the robust effect across all six models.
Do not rely on wording folklore. Negative-vs-positive framing is not significant for any model in aggregate.
Measure against the target agent stack. Model-specific behavior is large enough to change the best rule wording locally.
Ethics. No human subjects were studied. The experiment analyzes model-generated code under synthetic security prompts.
