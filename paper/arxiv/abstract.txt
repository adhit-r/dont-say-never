System-prompt rules are widely used to steer LLM coding agents away from insecure patterns. A popular heuristic, rooted in Wegner's ironic-process theory and reinforced by prompt-engineering folklore, holds that prohibition framing ("NEVER use eval()") activates the forbidden behavior, while positive alternatives ("Always use JSON.parse()") avoid this rebound. A 645-trial pilot motivated this prediction with one descriptive model-prompt anomaly.

We report a balanced 6-model replication across 3 OpenAI Codex/GPT models (GPT-5.4, GPT-5.4 Mini, GPT-5.3 Codex) and 3 Anthropic Claude models (Opus 4.6, Sonnet 4.6, Haiku 4.5). Across 6 vulnerability-eliciting prompts, 3 conditions, and 20 trials per cell, we collect 2,160 valid orchestration rows with zero final route errors. Here, "valid" means the model call completed and produced an analyzable code preview; it does not certify functional correctness.

We find: (1) rule injection reliably reduces detector-counted insecure API use in every model; control rates of 48-87% fall to 2-23% when the two rule conditions are pooled (Fisher's exact p < 0.001 in all 6 models); (2) positive framing has no consistent aggregate advantage; negative-vs-positive framing is not significant for any model in aggregate, and a random-effects equivalence test over 36 model-prompt strata estimates a positive-minus-negative risk difference of +1.2 percentage points (90% CI: -1.5 to +3.9), inside a pre-specified +/-5 percentage-point practical-equivalence margin; (3) the pilot's isolated prohibition backfire does not reproduce across the broader 36-cell replication.

We additionally run two extensions. A 1,080-trial non-API-naming extension shows that removing explicit insecure API names does not make all prompts inert: formula evaluation remains high-risk without naming eval() (85/120 control vulnerable), while hash and token prompts are 0/720 vulnerable without naming MD5 or Math.random(). A 2,160-row four-arm decomposition separates polarity from information content by comparing control, pure-negative, pure-positive, and combined rules. Combined rules are strongest overall (43/720 vulnerable, 6.0%), followed by pure-negative rules (75/720, 10.4%) and pure-positive rules (139/720, 19.3%).

We conclude that security rule presence and information content are robustly useful, but polarity is a model- and prompt-dependent second-order factor. The practical recommendation is simple: add targeted security rules for relevant CWE classes, prefer rules that state both the unsafe boundary and the safe replacement, and do not rely on Wegner-style "avoid never" heuristics without measuring them against the target agent stack.
